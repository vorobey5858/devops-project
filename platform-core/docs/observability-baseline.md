# Базовый слой наблюдаемости

## Сигналы по умолчанию

- Pod annotations по умолчанию открывают цель для Prometheus scrape.
- `ServiceMonitor` может включаться отдельно для каждого окружения через значения чарта.
- У каждого сервиса сохраняются стабильный `http` service port и стабильный путь к метрикам.
- Общие метки присутствуют на нагрузках, сервисах и мониторах, чтобы метрики, логи и alerts склеивались одинаково.

## Обязательные метрики

- пропускная способность запросов
- доля ошибок
- p95 latency
- перезапуски контейнеров
- сбои readiness
- сервисная метка `service`, одинаковая для метрик приложения, анализа rollout и reliability-запросов

## Контракт по меткам

- `app.kubernetes.io/name`
- `app.kubernetes.io/instance`
- `app.kubernetes.io/part-of=commerce-platform`
- `platform.example.io/environment`

Это минимальный контракт для:

- дашбордов Prometheus и Grafana
- фильтрации и корреляции в Loki
- SLO и incident-представлений слоя reliability
- анализа отката в Argo Rollouts

## Конвенция ServiceMonitor

- `serviceMonitor.enabled` в базовом чарте выключен и включается окруженческими overlays.
- `serviceMonitor.labels.release: kube-prometheus-stack` используется как базовое допущение для селектора Prometheus.
- `namespaceSelector.matchNames` указывает на namespace рабочей нагрузки, чтобы центральный стек мониторинга мог собирать метрики из прикладных namespace без ручных исключений.
- `targetLabels` и `podTargetLabels` переносят метки рабочей нагрузки в Prometheus, чтобы rollout/reliability слой видел единый контракт идентичности.

## Заглушки для дашбордов и оповещений

При онбординге сервиса должны быть зарезервированы заготовки дашбордов и оповещений для:

- availability по route или endpoint
- перцентилей задержки
- всплесков перезапусков
- сбоев probe
- насыщения ingress для внешних сервисов

## Допущения

- До включения `ServiceMonitor` уже установлен стек, совместимый с Prometheus Operator.
- Сервисы публикуют Prometheus-метрики на основном HTTP-порту; отдельный sidecar или отдельный порт для метрик пока не моделируются.
- Доставка логов, CRD для alert rules и SLO-определения будут расширяться другими слоями без изменения базового контракта по меткам.
- Stage-canary анализ использует те же метрики и метки, поэтому контракт наблюдаемости нельзя менять локально только в одном сервисе.
