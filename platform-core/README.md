# Платформенное ядро

Этот каталог задаёт общий контракт платформы. Здесь лежит всё, что нужно новому сервису, чтобы попасть в кластер одинаковым способом: namespace, GitOps-точка входа, шаблон манифестов и базовые правила по наблюдаемости.

## Что здесь лежит

- `kubernetes/argocd/project-platform.yaml` и `app-demo-services.yaml` для GitOps-входа
- `kubernetes/namespaces/*` для окружений и базовой namespace-модели
- `kubernetes/observability/*` для платформенной наблюдаемости
- `templates/service-template/*` как общий Helm-шаблон сервиса
- `docs/architecture.md`, `docs/onboarding.md`, `docs/observability-baseline.md` как описание контракта

## Как это используется

1. Сервис получает values и GitOps-application вместо собственного набора Kubernetes-манифестов.
2. ArgoCD синхронизирует нужное окружение в кластер.
3. Шаблон сервиса сразу делает его совместимым с rollout, метриками и admission-политиками.

## Что важно в этом слое

- Путь поставки задаётся один раз и потом переиспользуется.
- Совместимость с `secure-delivery` и `reliability-lab` считается обязательной частью шаблона, а не отдельной доработкой.
- Команды меняют параметры своих сервисов, а не изобретают свой способ деплоя.

## Что не входит в этот каталог

- код самих сервисов;
- бизнес-логика релиза;
- runbook-и, DR и chaos-сценарии;
- секреты и внешние учётные данные.
