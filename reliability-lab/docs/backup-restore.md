# Базовый подход к backup и restore

## Базовый сценарий Velero

`backup/velero-values.yaml` настраивает local-first путь резервного копирования:

- S3-совместимое хранилище через MinIO или аналогичный endpoint
- файловый backup через Velero node-agent
- backup каждые 15 минут для `demo-stage`
- более широкий ежедневный backup, который также захватывает часть общего платформенного состояния

Цель — сохранить демо-окружение практичным и проверяемым без зависимости от CSI-snapshot support.

## Допущения по MinIO и объектному хранилищу

- bucket `platform-backups` уже создан
- endpoint доступен по адресу `http://minio.minio.svc:9000`
- включён path-style access
- credentials передаются через secret `cloud-credentials`
- этот же bucket доступен и с DR-кластера
- TLS для первой итерации лаборатории необязателен; если его включать позже, управление trust chain остаётся за пределами этого каталога

## Что попадает в backup

- объекты namespace `demo-stage` и содержимое PVC
- выбранные общие платформенные ресурсы, нужные для восстановления демо-пути
- cluster-scoped ресурсы, на которые ссылаются сохраняемые namespace

## Процедура тестового восстановления

Restore-тесты стоит запускать минимум раз за спринт и перед тем, как заявлять готовность к DR.

1. Убедиться, что есть свежий backup со статусом `Completed`:
   `velero backup get`
2. Проверить выбранный backup и зафиксировать его timestamp:
   `velero backup describe <backup-name> --details`
3. Создать изолированное восстановление с remapping namespace:
   `velero restore create restore-demo-stage --from-backup <backup-name> --namespace-mappings demo-stage:demo-stage-restore`
4. Дождаться статуса `Completed` и состояния `Ready` у всех восстановленных pod.
5. Выполнить ручные проверки в восстановленном namespace:
   - `/health` и `/ready`
   - хотя бы один read и один write путь в базе данных
   - логи на предмет отсутствующих secret, image pull errors и проблем с mount PVC
6. Сравнить метрики и число перезапусков с эталонным здоровым состоянием.
7. После сбора артефактов удалить восстановленный namespace:
   `kubectl delete namespace demo-stage-restore`

## Доказательства успешного восстановления

Для каждого restore-теста нужно сохранить:

- имя backup и время его завершения
- имя restore и время его завершения
- список восстановленных pod и их readiness-состояние
- результат endpoint- и database-smoke-проверок
- все нерешённые предупреждения из `velero restore describe`

## Точки принятия решения

- Если последний успешный backup старше 15 минут, нужно зафиксировать нарушение RPO и эскалировать.
- Если restore завершился, но pod не проходят readiness, сначала нужно проверить конфигурацию приложения, а не сразу объявлять backup нерабочим.
- Если объектное хранилище недоступно с DR-площадки, failover нужно останавливать и рассматривать это как outage платформенной зависимости.
- Если два restore-теста подряд завершаются неудачей, нужно открывать инцидент и временно снимать заявления о готовности к DR.
