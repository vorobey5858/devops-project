# Модель угроз

## Защищаемые активы

- исходный код и история Git
- учётные данные CI и ключи подписи
- OCI-образы и прикреплённые attestations для SBOM
- Kubernetes-нагрузки, проходящие admission-проверки через Kyverno
- сигналы о здоровье релиза, которые используются для отката

## Основные угрозы

| Угроза | Влияние | Контроль |
| --- | --- | --- |
| Секреты попали в Git | утечка учётных данных и компрометация цепочки поставки | Gitleaks в CI и сохранение артефактов для разбора |
| Уязвимые или вредоносные зависимости внутри контейнерного образа | в релиз попадает образ с эксплуатационным риском | Trivy gate до подписи |
| Доставка неподписанного образа | в кластер попадает недоверенный артефакт | подпись Cosign, проверка публичным ключом и policy `require-signed-images` |
| Использование изменяемого тега `latest` | возможен дрейф образа после одобрения | policy `disallow-latest-tag` |
| Отсутствие ресурсных ограничений контейнера | эффект noisy neighbor и нестабильное поведение развёртывания | policy `require-workload-resources` |
| Слабый security context контейнера | риск повышения привилегий и выхода из контейнера | policy `require-non-root-security-context` |
| Плохой релиз продвигается без автоматической обратной связи | деградация для пользователей и медленный откат | анализ Argo Rollouts и пороги в Prometheus |

## Границы доверия

1. Рабочая станция разработчика -> GitHub
2. GitHub Actions runner -> OCI-реестр
3. OCI-реестр -> контур admission-проверок Kubernetes
4. Kubernetes-нагрузка -> Prometheus и контроллер отката

## Остаточные риски

- Текущий базовый контур не делает автоматическую ротацию ключей подписи.
- Политика `require-signed-images` рассчитана на образы из реестра и поэтому в локальном bootstrap включается отдельно.
- Текущий базовый контур пока не обновляет GitOps-манифесты автоматически, потому что этот контракт зависит от платформенного процесса развёртывания.

## Явные допущения

- Сервисные нагрузки сохраняют имя контейнера `app`, которое используется в примерах rollout и policy.
- Рендеринг `Rollout` в `service-template` сохраняет те же probes, resources, labels и поля `securityContext`, что и путь через `Deployment`.
- Пороговые значения SLO из `reliability-lab/docs/slo-sli.md`, которыми владеет reliability-команда, остаются источником для сигналов success-rate, latency и restart rollback.
