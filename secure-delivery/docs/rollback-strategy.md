# Стратегия отката

## Базовый сценарий

Базовый сценарий использует Argo Rollouts с канареечными шагами на 5%, 25%, 50% и 100%.
После каждой фазы продвижения идёт пауза и проверка метрик, а `progressDeadlineAbort: true` останавливает релиз, если развёртывание не может продвинуться дальше.
Основной демонстрационный сценарий привязан к namespace `demo-stage`, где сервисы уже рендерятся через `service-template` с включённым `rollout.enabled: true`.

## Условия автоматического отката

- доля успешных ответов падает ниже `95%`
- p95 задержка поднимается выше `500ms`
- число перезапусков контейнера становится больше `0` в окне наблюдения
- время продвижения развёртывания превышает `600s`

## Поток анализа

1. `Rollout` переводит небольшую долю трафика на canary ReplicaSet.
2. Шаблоны `success-rate` и `p95-latency` проверяют пользовательское поведение сервиса.
3. `restart-rate` добавляет техническую проверку состояния платформы перед дальнейшим расширением трафика.
4. Любая проваленная метрика помечает канареечный шаг как неуспешный, после чего Argo Rollouts прерывает релиз и возвращает стабильную ревизию.
5. `rollbackWindow.revisions: 3` хранит несколько последних ревизий для быстрого восстановления.

## Что делает оператор после автоматического отката

1. Проверяет Trivy, SBOM и `release-metadata` от сборки, которая выпустила этот образ.
2. Смотрит результаты анализа развёртывания и графики Prometheus для провалившегося канареечного шага.
3. Сверяется с соответствующим runbook в `reliability-lab/docs/runbooks.md`.
4. Исправляет проблему, пересобирает образ и продвигает новый неизменяемый тег вместо повторного использования неудачной версии.

## Явные допущения по совместимости

- Сервисы публикуют `http_requests_total` и `http_request_duration_seconds_bucket` либо совместимый с ними набор метрик в Prometheus.
- `service-template` сохраняет labels `app.kubernetes.io/name`, readiness, liveness, resources и контейнерный `securityContext`, поэтому манифест `Rollout` остаётся совместимым с платформенным контрактом.
- `analysis-templates.yaml` синхронизируется в `demo-stage` до появления stage-нагрузок с `Rollout`.
