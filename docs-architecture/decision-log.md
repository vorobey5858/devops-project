# Журнал решений

## ADR-001: один монорепозиторий вместо трёх отдельных репозиториев

Статус: принято

Почему:

- так проще синхронизировать архитектурные решения
- удобнее держать демо-поток и историю Git в одном месте
- проще проверять интеграционные контракты между слоями без дублирования артефактов

## ADR-002: GitHub Actions вместо GitLab CI

Статус: принято

Почему:

- шаблоны сценариев удобно хранить прямо в `.github`
- проект остаётся ближе к типичному public GitHub-сценарию
- не нужен отдельный self-hosted GitLab как обязательная зависимость

## ADR-003: `k3s` как базовый runtime

Статус: принято

Почему:

- `k3s` лучше подходит для лёгкого VPS-сценария и не требует отдельного control plane на базе Docker
- решение остаётся близким к реальному сценарию работы `k3s` и Kubernetes
- один и тот же runtime можно использовать и локально, и на недорогом Linux-хосте без смены GitOps-модели

## ADR-004: CloudNativePG как базовый HA-подход для PostgreSQL

Статус: принято

Почему:

- операторный подход проще поддерживать и объяснять
- легче показывать lifecycle, backup и failover
- решение хорошо ложится на Kubernetes-native сценарий
